# מצב הרפתקה — תכנון

> **Status: historical product concept.** This predates the current Adventure
> implementation. Use `AGENTS.md` and `GAME_TECH_GUIDE.md` for engineering
> rules; use `worlds.js`, `adventure.js`, and current tests for implemented
> behavior.

מסמך תכנון למצב משחק חדש ("הרפתקה") שחי **במקביל** למצב התפריטים הקיים.
במקום לבחור פעילות מתפריט, הילד חי בתוך עולם משחק: דמות אישית שהוא מעצב,
בסיס, מפת עולמות והתקדמות — כשכל הלמידה (רנדום ממושקל + פתיחה הדרגתית)
רצה מתחת למכסה.

רף האיכות (לפי הרפרנסים שסופקו): רקעים מצוירים מלאים, דמויות עם אנימציות
חיות, HUD משחקי, בן-לוויה, עולם שמרגיש נושם — לא "תפריט יפה".

**מה לא בתכולה:** קרבות. המפגשים בעולם הם משחקונים לימודיים.
**החלטה נדחית:** אילו משחקונים בדיוק (מהקיימים + חדשים) — נחליט בנפרד,
ואז נעצב לכל אחד סקין. התשתית כאן בנויה כך שמשחקון הוא "שקע" גנרי.

---

## 1. עקרונות הליבה

1. **מצב מקביל, לא החלפה.** מצב התפריטים הקיים נשאר כמו שהוא. מהמסך הראשי
   נכנסים ל"הרפתקה" — עולם נפרד עם ניווט משלו. הנתונים נשמרים ב-namespace
   נפרד (`adv_...`) כך שאין התנגשות עם ההתקדמות הקיימת.

2. **עולם = גוף ידע אחד.** כל עולם ממוקד בנושא אחד (למשל: האות א׳ על כל
   הניקודים שלה, צבעים באנגלית, ספרות 1–10). **כל** המשחקונים בתוך העולם
   קוראים ומעדכנים את **אותו מערך משקלים** — מה שהילד התקשה בו במשחקון
   אחד יחזור ויתורגל גם באחרים.

3. **מנגנון הלמידה לא משתנה.** רנדום ממושקל (`getWeightedRandomIndex`),
   עדכון משקלים (±1, טווח 0–15), פתיחה הדרגתית של `setItems` פריטים כשהכול
   מגיע ל-0 — הכול נשאר. רק ה"עטיפה" משתנה: במקום progress bar ומסך news,
   יש שביל הרפתקה, XP ודמות שמלמדת פריטים חדשים.

4. **יופי ותחושת חיים קודם כול.** רקעים מצוירים מלאים (בהמשך — ג'נרוט
   ב-OpenAI Images), דמויות שזזות כל הזמן (אנימציות idle), אפקטים,
   מעברים רכים. עד שלב האומנות עובדים עם placeholder-ים, אבל **הלייאאוט
   והאנימציות נבנים מראש למידות של רקע מלא**.

5. **הדמות שייכת לילד.** עיצוב דמות אישית הוא פיצ'ר ליבה, לא קישוט:
   הילד בוחר ומרכיב את הדמות שלו, והיא מלווה אותו בכל מסך.

## 2. חוויית המשחק — המסכים

### 2.1 עיצוב הדמות (Character Creator)

בכניסה ראשונה למצב הרפתקה — מסך יצירת דמות. הדמות בנויה **שכבות**:

```
בסיס (גוף/צבע עור) → תסרוקת + צבע → תלבושת → אקססורי (כובע/משקפיים/כנפיים) → בן-לוויה
```

- כל שכבה = ספרייט שקוף; ההרכבה ב-CSS (ערימת `<img>`/`background`), כך
  שכל שילוב עובד בלי לג'נרט תמונה פר שילוב.
- חלק מהפריטים פתוחים מההתחלה, השאר נקנים במטבעות (ראו 2.5) — זה מנוע
  המוטיבציה המרכזי.
- אפשר לחזור ולשנות בכל רגע מ"ארון החפצים" בבסיס.
- הדמות המורכבת מופיעה בכל מסך: בבסיס, על המפה, ליד המשחקונים, במסכי פרס.

### 2.2 הבסיס (Hub)

מסך פתיחה של המצב: חדר מצויר (ספרייה קסומה / ארמון חדי-קרן) ובו הדמות
ובן-הלוויה, חיים ונושמים (ריחוף עדין, מצמוץ, נרות מהבהבים ברקע). בחדר
"תחנות" לחיצות עם הילה זוהרת:

| תחנה | מה היא פותחת |
|------|--------------|
| **פורטל ההרפתקאות** | מפת העולמות (2.3) |
| **ספר הנושאים** | צפייה בעולמות וההתקדמות בכל אחד; קפיצה ישירה לעולם פתוח |
| **ארון החפצים** | עיצוב הדמות + קניית פריטים במטבעות |

בפינת המסך: כרטיס שחקן קבוע (אווטאר, רמה, פס XP, מטבעות).
בכניסה ראשונה — בן-הלוויה מוביל טוטוריאל קצר בבועות דיבור.

### 2.3 מפת העולמות

איים מרחפים בשמיים על שביל נקודות (בסגנון הרפרנס), עם עננים בתנועת
פרלקסה איטית וספינות אוויר שחולפות ברקע. כל אי = עולם ידע. אי נעול מציג
מנעול + תנאי פתיחה ("נפתח ברמה 5" / "סיימו את עולם הצבעים"). הדמות עומדת
על האי הנוכחי ו**הולכת** באנימציה לאי שנבחר. לחיצה על אי פתוח → מסך העולם.

### 2.4 מסך עולם — שביל המפגשים

בתוך עולם: רקע מצויר בנושא העולם ושביל של 6–10 תחנות ("מפגשים").
כל מפגש הוא צומת על השביל, משני סוגים:

- **מפגש למידה** — דמות ידידותית (NPC) "מלמדת" את הפריטים החדשים שנפתחו:
  מציגה אותם אחד-אחד עם הקראה. זה בדיוק מסך ה-news הקיים, בעטיפת סיפור.
- **משחקון** — אחד מסוגי המשחק (אילו — יוחלט בנפרד), בסקין של העולם,
  על מאגר הידע של העולם.
- **אתגר הסיום** — התחנה האחרונה: משחקון מסכם על *כל* פריטי העולם
  (ללא נעולים), עם חגיגה גדולה בסיום — העולם הושלם.

הדמות הולכת על השביל בין תחנות. כוכבים (1–3) לכל מפגש לפי דיוק
(מתוך `attemptHistory` של אותו סשן).

### 2.5 מטא-התקדמות (מה שגורם לזה להרגיש "משחק אמיתי")

| מערכת | תפקיד |
|--------|-------|
| **XP ורמות שחקן** | כל תשובה נכונה = XP. רמות פותחות איים במפה ("נפתח ברמה 10") |
| **מטבעות** | פרס על השלמת מפגשים; נקנים בהם פריטי עיצוב לדמות |
| **פריטי דמות** | תסרוקות/תלבושות/אקססוריז/בני-לוויה — התגמול המרכזי, בלי השפעה על קושי |
| **כוכבים** | איכות ביצוע פר מפגש; חלק מתנאי פתיחת עולמות |
| **בן-לוויה** | חד-קרן/דרקון קטן: מלווה, מעודד ("כל הכבוד!"), נותן רמזים אחרי טעות, מוביל טוטוריאל |

טעות במשחקון היא עדינה תמיד: אנימציית "אופס" קטנה, רמז מבן-הלוויה
(הקראה חוזרת / צמצום אפשרויות), והמשך — בלי מסכי כישלון ובלי איבוד התקדמות.

### 2.6 מסגרת המשחקון (Adventure Frame)

כל משחקון רץ בתוך "מסגרת הרפתקה" אחידה — זה השקע הגנרי:

```
+--------------------------------------------------+
| [כרטיס שחקן]                        [יציאה למפה] |
|                                                  |
|            << אזור המשחקון עצמו >>               |
|                                                  |
| [דמות + בן-לוויה]          [התקדמות המפגש 🟡🟡⚪] |
+--------------------------------------------------+
        רקע: איור מלא בנושא העולם
```

המסגרת מספקת לכל משחקון, בחינם: רקע העולם, הדמות ובן-הלוויה (עם תגובות
אנימציה להצלחה/טעות), רמזים, מד התקדמות המפגש, ומסך פרס בסיום (XP,
מטבעות, כוכבים, קונפטי). המשחקון עצמו אחראי רק על האינטראקציה הלימודית —
בדיוק כמו החוזה הקיים של `BaseGameComponent`.

## 3. מודל הידע המשותף — השינוי הטכני המרכזי

### 3.1 הבעיה בקיים

היום ה-namespace של המשקלים הוא ה-`currentAppId` של הפעילות
(`getWeightsKey` ב-[storage.js:72](storage.js#L72)): לכל משחק מערך משקלים
משלו, גם אם הוא על אותה רשימה. בהרפתקה צריך **מפתח אחד פר עולם**.

### 3.2 הפתרון: עולם וירטואלי כ"אפליקציה" עם מפתח ידע

קובץ חדש `worlds.js` מגדיר את העולמות:

```javascript
const WORLDS = [
  {
    id: 'aleph',                    // מפתח הידע: כל המפגשים כותבים אליו
    name: 'עולם האות א׳',
    unlock: {playerLevel: 1},       // או {world: 'colors', stars: 10}
    listName: 'HEBREW_LETTERS_WITH_NIKUD',
    itemFilter: {field: 'letter', value: 'א'},  // חיתוך רשימה קיימת (3.3)
    setItems: 3,                    // כמה פריטים נפתחים בכל "מפגש למידה"
    art: {island: '...', bg: '...'},
    encounters: [
      {type: 'learn'},
      {type: 'game', game: '<appType>', questionIndex: '...', resultIndex: '...', goal: 6},
      {type: 'game', game: '<appType>', /* ... */},
      {type: 'learn'},
      // ...
      {type: 'final', game: '<appType>', practiceAll: true},
    ],
  },
  // ...
];
```

`game` הוא `appType` — קיים או חדש. אילו משחקונים ייכנסו לכל עולם יוחלט
בנפרד; הסכמה לא תלויה בזה. `goal` = כמה הצלחות משלימות את המפגש.

נקודות מימוש:

- **ניתוב:** נתיבים חדשים `/adventure` (בסיס), `/adventure/map`,
  `/adventure/world/:worldId`, `/adventure/play/:worldId/:encounterIndex`.
- **מפתח משקלים:** כל קריאות ה-API של הלמידה מקבלות `key`. במפגשי הרפתקה
  מעבירים `adv_<worldId>` במקום `currentAppId` — כך **כל** המפגשים בעולם
  חולקים משקלים, היסטוריה והתקדמות. נדרש שינוי קטן: לאפשר לרכיבי המשחק
  הקיימים לקבל את המפתח וההגדרות מבחוץ (ראו 3.4) במקום מ-`getItemById`.
- **אחסון דרך `storage.js` בלבד** — כל מצב ההרפתקה (XP, רמה, מטבעות,
  כוכבים, פריטי דמות, עולם נוכחי) נשמר במפתחות `adv_player`, `adv_avatar`,
  `adv_stars` וכו', ולכן מסונכרן אוטומטית ל-Firebase פר משתמש, בחינם.

### 3.3 `itemFilter` — חיתוך רשימות קיימות

עולם "האות א׳" לא מצריך רשימה חדשה: `HEBREW_LETTERS_WITH_NIKUD` (88 פריטים)
כבר מכילה את כל הווריאציות. נוסיף פונקציה `getWorldItems(world)` שמחזירה
תת-רשימה מסוננת (וממופה לאינדקסים המקוריים). המשקלים נשמרים על התת-רשימה.
עולמות אחרים (צבעים, מספרים) פשוט מפנים לרשימה מלאה בלי פילטר.

### 3.4 שימוש חוזר ברכיבי המשחק הקיימים

כדי לא לשכפל רכיבים כשמשחקון קיים נבחר לעולם:

1. נחלץ מהם את התלות ב-route: היום הם קוראים `currentAppId` מה-route
   ו-`getItemById(apps, id)` ([tester.js:355](tester.js#L355)). נוסיף נקודת
   הרחבה אחת: אם ה-id מתחיל ב-`adv_`, פונקציית resolve חדשה תחזיר "הגדרת
   אפליקציה וירטואלית" שנבנית מ-`worlds.js` (listName, אינדקסים, setItems,
   ומפתח המשקלים של העולם). **שינוי בנקודה אחת, אפס שינוי בתוך המשחקים.**
2. הסקין של כל משחקון (התאמה ויזואלית לעולם) — פרויקט עיצוב נפרד פר
   משחקון, אחרי שנחליט אילו משחקונים נכנסים. מסגרת ההרפתקה (2.6) נותנת
   בסיס אחיד עוד לפני הסקין הפרטני.

### 3.5 Seeding — עולם חדש יורש את מה שהילד כבר יודע ✅ (מומש)

ידע הוא פר עולם, אבל באתחול ראשון של עולם המערכת סורקת **את כל מה שהילד
שיחק אי-פעם על אותם פריטים** — עולמות הרפתקה אחרים על אותה רשימה וכל
משחקי התפריטים הרגילים שמשתמשים בה (בשני מצבי הפעילות), כולל היסטוריית
ניסיונות — ומצליבה לפי האינדקס ברשימת המקור:

- פריט עם ראיות נפתח מיד, במשקל לפי קושי: `2 + round(3 × difficulty)`
  כאשר difficulty = המקסימום בין (משקל מרבי קיים / 15) לשיעור הכישלונות
  בהיסטוריה. כלומר: נשלט → 2 (חזרה קלה), מתקשה → עד 5 (כמו חדש).
  **מה שקשה לילד מתועדף וחוזר יותר.**
- פריט בלי שום ראיה נשאר נעול ונפתח בהדרגה כרגיל (והוא שיוצג כ"חדש").
- הזריעה רצה רק במצב למידה ורק לעולמות הרפתקה; אתחול משחקי התפריטים
  לא השתנה.

### 3.6 מיפוי מנגנון השלבים הקיים לעולם

המכניקה הקיימת של פתיחה הדרגתית ממופה 1:1 לשביל:

| קיים היום | בהרפתקה |
|-----------|---------|
| `getWeightsForKey` מאתחל `setItems` ראשונים | "מפגש למידה" ראשון מציג אותם |
| `allZero` → פתיחת פריטים חדשים + `_new_items` | סיום מקטע בשביל → "מפגש למידה" הבא |
| מסך `/display/news` | NPC שמלמד את הפריטים החדשים |
| level complete (אין נעולים, הכול 0) | אתגר הסיום נפתח; השלמתו = סיום העולם |
| `attemptHistory` (5 ניסיונות) | חישוב כוכבים + בחירת רמזים |

## 4. עיצוב, אנימציה ואומנות

### 4.1 שפה חזותית

- **סגנון:** פנטזיה צ'יבי רכה — צבעי פסטל, קסם, קשתות, איים מרחפים.
  בן-לוויה ברירת מחדל: **חד-קרן קטן** 🦄. *קו האופי הסופי — בשלב האומנות.*
- **מבנה מסך אחיד:** רקע מצויר מלא (cover), שכבת ספרייטים (דמויות),
  שכבת UI (HUD, בועות, כפתורים) ב-HTML/CSS מעוגל ורך.
- **טיפוגרפיה:** פונט עברי עגול וידידותי (למשל Rubik / Varela Round).

### 4.2 אנימציה — "העולם חי" (דרישת רף מרכזית)

שום דבר על המסך לא קפוא. ספריית אנימציות CSS אחת (`adventure.css`) עם
קלאסים לשימוש חוזר:

| אנימציה | איפה |
|---------|------|
| ריחוף/נשימה (idle bob) | דמות, בן-לוויה, NPC, איים |
| הליכה (סיבוב + תזוזה) | דמות על המפה ועל שביל העולם |
| פעימת הילה | כל אלמנט לחיץ (תחנות, איים, תשובות) |
| פרלקסה איטית | עננים, ספינות אוויר, חלקיקי קסם ברקע |
| קפיצת שמחה + לבבות | הדמות ובן-הלוויה אחרי הצלחה |
| "אופס" (ניעור ראש) | אחרי טעות — עדין, לא מעניש |
| קונפטי + כוכבים | מסכי פרס וסיום עולם |
| מעברי מסך | fade/slide רך בין בסיס↔מפה↔עולם↔משחקון |

מימוש: CSS animations/transitions + ספרייט שקוף פר פוזה (2–3 פוזות
לדמות). בלי מנוע משחק — DOM בלבד, כמו ברפרנס.

- **סאונד:** אפקטים קיימים + מוזיקת רקע שקטה פר עולם (עם כפתור השתקה).

### 4.3 צנרת נכסים (Asset pipeline) — לשלב מאוחר יותר, אבל מוגדר עכשיו

כל הנכסים יישבו ב-`assets/adventure/` עם **מניפסט** (`art_manifest.json`):
לכל נכס — שם קובץ, תיאור, ופרומפט ג'נרוט. כשנגיע לשלב התמונות (OpenAI
Images), מריצים סקריפט שמייצר את החסרים. עד אז — placeholder-ים.

רשימת נכסים ראשונית:

| קטגוריה | נכסים | כמות משוערת |
|----------|-------|--------------|
| רקעים | בסיס, מפת שמיים, רקע פר עולם | ~8 |
| איים | ספרייט אי פר עולם (פתוח/נעול) | ~8 |
| דמות (שכבות) | גופים, תסרוקות, תלבושות, אקססוריז — פר פוזה | ~20 |
| דמויות משנה | בן-לוויה (2–3 פוזות), NPC מלמד פר עולם | ~8 |
| UI | מסגרות HUD, מטבע, כוכב, מנעול, כפתורים, בועות דיבור | ~10 |

עיקרון חשוב: **המשחק שלם ופלייבילי עם placeholder-ים** — האומנות מחליפה
קבצים, לא קוד. שכבות הדמות מוגדרות כך שכל שילוב עובד בהרכבה ב-CSS.
